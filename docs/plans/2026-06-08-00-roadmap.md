## Incremento 1: Fundaciones ✅ COMPLETADO

**Plan detallado:** `docs/plans/2026-06-08-01-fundaciones.md`

Estructura de la solución con las 4 capas (Domain / Application / Infrastructure /
Presentation), app Avalonia que abre una ventana, SQLite cableado vía EF Core con
`AppDbContext` mínimo, y los cuatro proyectos xUnit con el ciclo TDD operativo. No
contiene ninguna feature funcional: es el andamiaje sobre el que se construye todo.

**Dependencias:** ninguna (punto de partida).

---

## Incremento 2: Dominio + Persistencia + Backups ✅ COMPLETADO

**Plan detallado:** `docs/plans/2026-06-08-02-dominio-persistencia.md`

Entidades del dominio (Usuario, Producto, Categoria, Proveedor, UnidadMedida,
MovimientoStock, LogAuditoria y sus enums), DbContext con Fluent API, primera migración
real que crea el esquema completo, base de datos ubicada en el directorio de datos del
usuario (no junto al ejecutable), inyección de dependencias cableada en el arranque de
Avalonia, y el servicio de backup (pre-migración + periódico cada 12 h con retención
7 días y salvaguarda de conservar al menos el más reciente).

**Dependencias:** Inc 1.

**Resultado:** implementado con **42 tests verdes** (Dominio + Persistencia + Backups).

---

## Incremento 3: Auth + Usuarios + Autorización ✅ COMPLETADO

Login con BCrypt (hash + sal, nunca texto plano), sesión en memoria mientras la app
corre con soporte de cierre/cambio de sesión sin reiniciar. Asistente de primer
arranque para crear el Admin inicial (no hay contraseña por defecto). ABM de usuarios
solo para Admin: alta, edición, baja lógica (`Activo = false`). Autorización verificada
en la capa Application antes de ejecutar cualquier acción sensible; la UI refleja el
estado pero no es la frontera de seguridad.

**Dependencias:** Inc 2. Va antes del catálogo: la seguridad es el cimiento sobre el
que se construyen las features operativas.

**Resultado:** implementado con **128 tests verdes** (Domain 6 / Application 38 /
Infrastructure 56 / Presentation 28). Incluye UI de login y asistente de primer
arranque. Pasó revisión de seguridad.

## Incremento 4: Catálogo

ABM completo de productos (con SKU, código de barras opcional, precio costo y venta,
stock mínimo, baja lógica), categorías, proveedores y unidades de medida (tabla
editable, no enum). Búsqueda de productos por SKU, código de barras y nombre.
Auditoría automática de cambios de precio y de altas/bajas de productos (registra en
`LogAuditoria` con el usuario que hizo el cambio).

**Dependencias:** Inc 3 (el catálogo requiere usuario autenticado y autorización).

---

## Incremento 5: Movimientos de stock

Registro transaccional ACID de entradas y salidas: en la misma transacción se inserta
el `MovimientoStock` y se actualiza `StockActual` en `Producto`, con los motivos
Compra / Venta / Ajuste / Merma. Validación de stock no negativo (advertir con
confirmación, no bloquear). Función de recálculo de stock desde los movimientos como
red de seguridad ante descuadres.

**Dependencias:** Inc 4 (los movimientos operan sobre el catálogo).

---

## Incremento 6: Reportes + Auditoría

Cinco reportes solo para Admin, todos exportables a CSV: valorización de stock (a
costo y a venta), stock por categoría, historial de movimientos por producto
(filtrable por rango de fechas), productos más movidos, y visor del log de auditoría
(filtrable por usuario y rango de fechas). Exportación a CSV desde la capa Application,
sin dependencias de UI.

**Dependencias:** Inc 5 (los reportes consultan movimientos y stock real).

---

## Incremento 7: Distribución

Empaquetado con Velopack como self-contained (runtime .NET 10 incluido): `Setup.exe`
en Windows y `AppImage` en Linux. Actualizador in-app con fallback dominio propio →
GitHub público y UX por severidad (normal / important / critical con modo degradado).
Licenciamiento: licencia offline firmada con criptografía asimétrica, atada a la
máquina (machine fingerprint cross-platform), pantalla de bloqueo sin licencia válida,
y herramienta CLI interna (nunca distribuida) para generar licencias. También incluye
el **token de reset de Admin firmado** (último recurso de recuperación de acceso, spec
§5.1): reutiliza la misma infraestructura criptográfica asimétrica del licenciamiento.

**Dependencias:** Inc 6. Va al final: se empaqueta y licencia un producto terminado,
no un andamiaje.
