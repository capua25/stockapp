**Goal:** Construir el catálogo completo del sistema — gestión de Productos (uso diario, Admin +

## Propuesta

### Contexto y motivación

El Inc 2 dejó las entidades del catálogo creadas y persistidas (`Producto`, `Categoria`,
`Proveedor`, `UnidadMedida`) con su esquema base y la migración `InitialCreate`. El Inc 3 construyó
toda la infraestructura transversal de seguridad: `IAuthorizationService.Verificar(rol, accion)`,
el catálogo de `Permisos`, la auditoría (`IAuditLogger.RegistrarAsync` + `LogAuditoria`), la sesión
(`ICurrentSession`) y los patrones de servicio/repositorio/ViewModel.

El Inc 4 es el **primer incremento de negocio con UI funcional de uso diario**: le da al usuario las
pantallas para dar de alta, editar, buscar y dar de baja productos, y a la administración las pantallas
para mantener las tablas maestras de las que dependen los productos. No introduce movimientos de stock
(Inc 5) ni reportes (Inc 6): solo el ABM del catálogo y su navegación.

Se apoya en:
- **Inc 2** — entidades + esquema + `InitialCreate`. `Producto` ya está completo para el alcance.
- **Inc 3** — `IAuthorizationService`, `Permisos`, `IAuditLogger`/`LogAuditoria`, `ICurrentSession`,
  patrones de `UsuarioService`/`UsuarioRepository`/ViewModels y el `ViewLocator` por convención.

### Objetivos medibles

1. ABM completo (alta / edición / baja lógica / listado / búsqueda) de **Producto**, accesible a
   Admin y Operador.
2. ABM completo de **Categoria**, **Proveedor** y **UnidadMedida**, accesible **solo a Admin**.
3. Autorización efectiva por capa Application: un Operador que intente gestionar tablas maestras
   recibe `UnauthorizedAccessException` antes de ejecutar el caso de uso.
4. Toda alta / baja / modificación del catálogo genera un `LogAuditoria` con detalle granular por
   campo (`campo: viejo → nuevo`).
5. Unicidad garantizada por BD: `Proveedor.Nombre`, `UnidadMedida.Nombre`, `UnidadMedida.Abreviatura`
   (más `Categoria.Nombre` que ya es único desde Inc 2).
6. Navegación por menú lateral con pantallas separadas; "Productos" visible para ambos roles, grupo
   "Tablas maestras" visible solo para Admin.
7. `dotnet build StockApp.sln` y `dotnet test` pasan con 0 errores y **sin regresiones** en los
   tests del Inc 3.

### Alcance

#### Entra (In Scope)

- **Domain:** baja lógica (`Activo`) en `Categoria` y `Proveedor`; extensión del enum
  `AccionAuditada` con los valores de catálogo.
- **Application:** 4 interfaces de repositorio + 4 servicios de aplicación (`ProductoService`,
  `CategoriaService`, `ProveedorService`, `UnidadMedidaService`), cada uno con autorización +
  auditoría granular; rediseño del modelo de permisos (separar Productos de Tablas Maestras).
- **Infrastructure:** 4 repositorios EF Core; migración `AddCatalogoExtensions` (columnas `Activo`
  + índices únicos nuevos).
- **Presentation:** 4 pantallas (List/Form por entidad) con sus ViewModels y Views; menú lateral
  con visibilidad por rol; un servicio/factory de navegación; cableado de DI.
- Tests por capa siguiendo TDD estricto.

#### Fuera (Out of Scope)

- **Movimientos de stock** (entradas/salidas/ajustes, recálculo de existencias) → **Inc 5**.
- **Reportes** (stock bajo, valorización, listados imprimibles) → **Inc 6**.
- Token de reset firmado / criptografía de licenciamiento → **Inc 7**.
- Importación/exportación masiva de catálogo (CSV/Excel) — no planificada en este incremento.
- Borrado físico de entidades del catálogo: se usa baja lógica, no `Delete`.

### Decisiones firmes (tomadas por el usuario — no se re-discuten)

> Estas 7 decisiones son requisitos de entrada del incremento. El diseño se construye sobre ellas.

**D1 — UI: pantallas separadas con menú lateral (NO tabs).**
"Productos" es ítem de menú de primer nivel para Admin y Operador (uso diario). El grupo
"Tablas maestras" (Categorías / Proveedores / Unidades de medida) es visible **solo para Admin**.
*Justificación:* el modelo de UI refleja el modelo de permisos y escala con Inc 5 (Movimientos) e
Inc 6 (Reportes), que también irán al menú lateral. Se introduce un **servicio/factory de
navegación** en Presentation en vez de inflar el constructor del `ShellViewModel` con 4+
dependencias nuevas.

**D2 — Permisos: Operador solo gestiona Productos; Tablas Maestras solo Admin.**
Se **rediseña** el permiso único `GestionarCatalogo` (hoy asignado a Operador) en dos permisos
nuevos: `GestionarProductos` (Admin + Operador) y `GestionarTablasMaestras` (solo Admin).
*Estrategia de migración sin romper Inc 3:* `GestionarCatalogo` se reemplaza por los dos nuevos en
`Permisos.cs`; `AuthorizationService.AccionesOperador` pasa de `{GestionarCatalogo, ...}` a
`{GestionarProductos, ...}`; los 3 usos en `AuthorizationServiceTests.cs` (líneas 16, 27, 53) se
actualizan a los permisos nuevos como parte del primer commit del incremento. No hay otros consumidores
del permiso (verificado por búsqueda en el código). *Justificación:* el modelo de permisos debe
reflejar que las tablas maestras son configuración administrativa, no operación diaria.

**D3 — Unicidad (índices únicos nuevos en la migración).**
`Proveedor.Nombre` único; `UnidadMedida.Nombre` único; `UnidadMedida.Abreviatura` única.
(`Categoria.Nombre` ya es único desde Inc 2.) La unicidad se valida también en el servicio
(excepción de dominio amigable) además del índice de BD como red de seguridad.

**D4 — Auditoría: todo el catálogo.**
Se extiende `AccionAuditada` con valores nuevos para altas/bajas/modificaciones de `Categoria`,
`Proveedor` y `UnidadMedida`, más `ModificacionProducto` para cambios que no son de precio (ya
existen `CambioPrecio`, `AltaProducto`, `BajaProducto`). **Los valores nuevos van SIEMPRE al final
del enum** (es `int` en BD — el enum actual ocupa los índices 0–6, los nuevos arrancan en 7); no se
reordenan los existentes.

**D5 — Baja lógica con `Activo` (bool) en `Categoria`, `Proveedor` y `UnidadMedida`.**
Replica el patrón de `Producto`/`Usuario` en las tres tablas maestras. Requiere columna nueva → migración.
`UnidadMedidaService.BajaLogicaAsync` hace `Activo=false`, no borrado físico. Nunca se borra
físicamente ninguna entidad del catálogo.

**D6 — Auditoría granular por campo.**
Mismo patrón que el Inc 3 usó para cambio de rol: `Detalle` con formato `"campo: viejo → nuevo"`,
un fragmento por campo modificado.

**D7 — Migración nueva `AddCatalogoExtensions`.**
Cubre: columna `Activo` en `Categorias` y `Proveedores`, e índices únicos de D3. **Riesgo verificado:**
SQLite + EF Core con `ALTER TABLE ADD COLUMN ... DEFAULT` tiene limitaciones; el diseño/apply debe
verificar cómo genera EF 10 esta migración para SQLite (default constante vs. recreación de tabla
mediante el patrón de "table rebuild" que EF aplica en SQLite).

## Especificación

> Resumen de requisitos y escenarios clave de los 5 dominios. Spec completa en engram `sdd/catalogo/spec`.

### gestion-productos (Admin + Operador)

**Alta:** `Codigo` (SKU) único en activos e inactivos; `CodigoBarras` único si se provee; `PrecioCosto`/`PrecioVenta ≥ 0`; `UnidadMedidaId` debe existir. Registra `AltaProducto`.

**Edición:** `Codigo` no se modifica post-alta. Auditoría granular por campo: `"campo: viejo → nuevo"` — un registro por campo cambiado. Cambios de precio usan `CambioPrecio`; resto usa `ModificacionProducto`. Sin cambios → sin log.

**Baja lógica:** `Activo=false`. No existe borrado físico. Producto ya inactivo → `InvalidOperationException`.

**Búsqueda:** por SKU exacto, código de barras exacto o nombre parcial (LIKE, case-insensitive). Solo activos por defecto; flag explícito incluye inactivos.

**Autorización:** `Permisos.GestionarProductos` — Admin y Operador.

Escenarios clave:
- Alta con SKU duplicado → `InvalidOperationException`
- Precio negativo → `ArgumentException` antes de persistir
- Edición con campos sin cambio → cero entradas de auditoría
- Operador → no lanza `UnauthorizedAccessException` (control cruzado con tablas maestras)

---

### gestion-tablas-maestras (solo Admin)

**Categorías:** `Nombre` único (activas + inactivas). Alta/Edición/Baja lógica (`Activo`). Registra `AltaCategoria`, `ModificacionCategoria`, `BajaCategoria`. Baja de categoría con productos asociados: los productos no se modifican (FK no es SET NULL en baja lógica).

**Proveedores:** `Nombre` único. Campos opcionales: `Telefono`, `Email`, `Direccion`, `Notas`. Alta/Edición/Baja lógica. Registra `AltaProveedor`, `ModificacionProveedor`, `BajaProveedor`.

**Unidades de medida:** `Nombre` único + `Abreviatura` única. Alta/Edición/Baja lógica (`Activo=false`). Registra `AltaUnidadMedida`, `ModificacionUnidadMedida`, `BajaUnidadMedida`.

**Autorización:** `Permisos.GestionarTablasMaestras` — solo Admin. Operador → `UnauthorizedAccessException`.

Escenarios clave:
- Nombre duplicado en cualquier tabla maestra → `InvalidOperationException`
- `UnidadMedida.Abreviatura` duplicada → `InvalidOperationException`
- Operador intenta crear categoría → `UnauthorizedAccessException`
- Edición con auditoría granular: `"Nombre: viejo → nuevo"`

### Migración AddCatalogoExtensions (D-MIG)

Comando: `dotnet ef migrations add AddCatalogoExtensions` (dotnet-ef 10.0.8, SDK `/home/capua25/.dotnet`).

El `Up()` generado debe contener:
- `AddColumn("Categorias", "Activo", defaultValue: true)` — sin table-rebuild
- `AddColumn("Proveedores", "Activo", defaultValue: true)`
- `AddColumn("UnidadesMedida", "Activo", defaultValue: true)`
- `CreateIndex("IX_Proveedores_Nombre", unique: true)`
- `CreateIndex("IX_UnidadesMedida_Nombre", unique: true)`
- `CreateIndex("IX_UnidadesMedida_Abreviatura", unique: true)`

EF Core 10 + SQLite soporta `ALTER TABLE ADD COLUMN DEFAULT <constante>` y `CREATE UNIQUE INDEX` sin table-rebuild. Filas preexistentes quedan con `Activo=true` garantizado.

**Test de migración**: usa SQLite archivo temporal + `MigrateAsync` (NO in-memory / EnsureCreated — estos saltean migraciones). Patrón `InitialCreateMigrationTests`.

### Búsqueda BuscarAsync (D-SEARCH)

var q = _ctx.Productos.AsQueryable();
if (!string.IsNullOrWhiteSpace(sku))          q = q.Where(p => p.Codigo.Contains(sku));

return await q.Where(p => p.Activo).OrderBy(p => p.Nombre).ToListAsync();
