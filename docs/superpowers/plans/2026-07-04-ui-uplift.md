# StockApp UI Kit — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Elevar la UX/UI de StockApp a un aspecto profesional tipo dashboard/SaaS mediante una capa de diseño aditiva sobre FluentTheme, aplicada globalmente.

**Architecture:** Capa de ResourceDictionaries en src/StockApp.Presentation/Themes/ (Tokens, Typography, Controls) incluidos en App.axaml DESPUÉS de FluentTheme para overridear. No se reemplaza Fluent; se le agregan tokens de color, fuente Inter global, estilos globales de controles, sidebar con estado activo e iconos. Reversible quitando includes.

**Tech Stack:** Avalonia 12.0.4, .NET 10, CommunityToolkit.Mvvm, Optris.Icons.Avalonia (nueva dependencia — ver nota de desviación en Tarea 3), fuente Inter (ya en el csproj).

## Global Constraints
- Capa ADITIVA: nunca reemplazar FluentTheme; los ResourceDictionaries van incluidos DESPUÉS de FluentTheme en App.axaml.
- Paleta (recursos nombrados, único lugar de verdad): Brand #2563EB, Fondo #F8FAFC, Superficie #FFFFFF, Borde #E2E8F0, TextoPrimario #0F172A, TextoSecundario #64748B, Success #16A34A, Warning #D97706, Danger #DC2626, Info #0EA5E9. Además, variantes derivadas necesarias para estados de interacción (hover/pressed/disabled): BrandHover #1D4ED8, BrandPressed #1E40AF, BrandSuave #EFF6FF, DangerHover #B91C1C, DangerPressed #991B1B, DeshabilitadoFondo #F1F5F9, DeshabilitadoTexto #94A3B8, DeshabilitadoBorde #E2E8F0. Nombres de recurso (Brush) exactos y usados consistentemente en todas las tareas: `BrandBrush`, `BrandHoverBrush`, `BrandPressedBrush`, `BrandSuaveBrush`, `FondoBrush`, `SuperficieBrush`, `BordeBrush`, `TextoPrimarioBrush`, `TextoSecundarioBrush`, `SuccessBrush`, `WarningBrush`, `DangerBrush`, `DangerHoverBrush`, `DangerPressedBrush`, `InfoBrush`, `DeshabilitadoFondoBrush`, `DeshabilitadoTextoBrush`, `DeshabilitadoBordeBrush`.
- Tipografía Inter global + escala: título de vista 20/SemiBold (clase `titulo-vista`), sección 16/Medium (clase `seccion`), body 14 (clase `body`), caption 12 (clase `caption`).
- Conventional commits, sin Co-Authored-By ni atribución de IA. Commits frecuentes (uno por tarea).
- No se toca lógica de negocio. La suite de tests (`dotnet test StockApp.sln`) debe quedar verde.
- Validación de estilos = build + correr la app + inspección visual (NO unit tests de apariencia). Solo la lógica del quick win de Enter (Tarea 6) y el estado activo del sidebar (Tarea 4) llevan test real, porque son las únicas piezas de este plan con lógica de C# verificable sin infraestructura de UI headless (el repo no tiene Avalonia.Headless referenciado en `tests/StockApp.Presentation.Tests`).
- **Hallazgo verificado durante la investigación de este plan** (ver Tarea 1 y Tarea 6 para el detalle): `Program.cs` ya llama `.WithInterFont()`, por lo que Inter YA se renderiza hoy en toda la app aunque ningún XAML lo declare explícitamente. Y el botón "Entrar" de LoginView ya tiene `IsDefault="True"`, pero esto NO alcanza para que Enter dispare el login porque Avalonia's TextBox intercepta `Key.Enter` internamente antes de que llegue al mecanismo de botón-por-defecto (bug conocido, no específico de este repo — ver referencias en Tarea 6). Estos dos hallazgos ajustan el alcance real de las Tareas 1 y 6 respecto al diagnóstico original de la spec.

En teoría, eso debería disparar el comando al presionar Enter. **En la práctica no funciona** porque `TextBox` de Avalonia intercepta y marca `Handled` el evento `Key.Enter` internamente sin importar `AcceptsReturn`, por lo que nunca llega al mecanismo de "botón por defecto" de la ventana mientras el foco esté en un `TextBox` — que es exactamente el caso de los campos Usuario/Contraseña. Esto es un comportamiento documentado de Avalonia, no un bug de este repo (ver AvaloniaUI/Avalonia#860 y la discusión AvaloniaUI/Avalonia#17095 sobre "Enter and ESC keys in windows - best practices"). Por eso el `IsDefault="True"` existente es efectivamente inerte hoy, y el diagnóstico de la spec ("Enter no submitea, pendiente UX previo conocido") es correcto.
